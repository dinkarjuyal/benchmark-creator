def _url_from_selector(sel):
    # type: (parsel.Selector) -> str
    if isinstance(sel.root, str):
        # e.g. ::attr(href) result
        return strip_html5_whitespace(sel.root)
    if not hasattr(sel.root, "tag"):
        raise _InvalidSelector(f"Unsupported selector: {sel}")
    if sel.root.tag not in ("a", "link"):
        raise _InvalidSelector(

    return strip_html5_whitespace(href)

# Fix Summary: TextResponse.text Encoding Issue

## Problem
The `TextResponse.text` property was hardcoding the charset to UTF-8, ignoring 
the response's declared encoding. This caused responses with other charsets 
(like Latin-1, Windows-1252, etc.) to be incorrectly decoded, resulting in 
corrupted text with replacement characters (U+FFFD).

### Example of the Bug
```python
# Latin-1 encoded response
response = TextResponse(
    "http://example.com",
    body=b"caf\xe9",  # Latin-1 bytes for "café"
    headers={"Content-Type": ["text/html; charset=iso-8859-1"]}
)

# Before fix: response.text = "caf\ufffd"  (corrupted with replacement char)
# After fix:  response.text = "café"      (correct)
```

## Root Cause
In `scrapy/http/response/text.py`, the `text` property had this code:
```python
@property
def text(self) -> str:
    """Body as unicode"""
    benc = self.encoding
    if self._cached_ubody is None:
        charset = "charset=utf-8"  # BUG: hardcoded UTF-8, ignored response 
encoding
        self._cached_ubody = html_to_unicode(charset, self.body)[1]
    return self._cached_ubody
```

The `charset` variable was hardcoded to `"charset=utf-8"` instead of using the 
detected encoding in `benc`.

## Solution
Changed line 94 from:
```python
charset = "charset=utf-8"  # BUG: ignores response encoding
```

To:
```python
charset = f"charset={benc}" if benc else None
```

This ensures that the detected encoding (stored in `benc`) is used when decoding
the response body.
